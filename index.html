Overview

This lab builds on "Build Multi-Agent Systems with ADK," where participants previously constructed a Course Creation System with four specialized agents:

  • Researcher Agent: Uses Google Search for current information (modified to use Wikipedia Search in this lab to enable inspection of custom tool call tracing and evaluation)
  • Judge Agent: Critiques research quality and completeness
  • Content Builder Agent: Structures research into courses
  • Orchestrator Agent: Manages workflow and inter-agent communication

The system includes a web application for course requests. The Researcher, Judge, and Content Builder agents deploy as A2A agents in separate Cloud Run services, while the Orchestrator runs as another Cloud Run service with ADK Service API.

💡
The core challenge: How do you verify a distributed multi-agent system actually works well?

This lab replaces subjective "vibe checks" with data-driven assessment using the Vertex AI Gen AI Evaluation Service. Participants will implement Adaptive Rubrics and Tool Use Quality metrics to evaluate the multi-agent system, then automate this within a CI/CD pipeline.

Learning Objectives

You will build a Continuous Evaluation Pipeline to learn:

  1. Deploy agents to private tagged revisions in Google Cloud Run (shadow deployment)
  2. Run automated evaluation suites against specific revisions using Vertex AI Gen AI Evaluation Service
  3. Visualize and analyze results
  4. Integrate evaluation into CI/CD pipelines

Core Concepts: Agent Evaluation Theory

Two Assessment Types

Offline Experimentation and Continuous Evaluation with Automated Regression Testing serve different purposes:

Offline Experimentation Continuous Evaluation
Purpose Creative optimization process Verification process
Inputs Open-ended, variable Rigid, repetitive "golden" datasets
Goal Raise agent capability ceiling Prevent performance floor collapse
⚡
"Offline Experimentation" raises the ceiling of what agents can do; "Continuous Evaluation" prevents the floor from collapsing.

This lab focuses on Continuous Evaluation, developing an Automated Regression Testing Pipeline that runs automatically with code changes, similar to unit tests.

The "Vibe Check" Trap

Manual agent testing through conversation has significant limitations:

  • Non-Determinism: Agents answer differently each time; statistical significance requires large sample sizes
  • Invisible Regressions: Improving one prompt may break other use cases
  • Human Bias: "It looks good" lacks objectivity
  • Time-consuming: Manual testing of dozens of scenarios with every commit is inefficient

Two Grading Approaches

Code-Based Graders (Deterministic)

  • Measure strict constraints: "Did it return valid JSON?" "Did it call search tool?"
  • Advantages: Fast, cheap, 100% accurate
  • Limitations: Cannot judge nuance or quality

Model-Based Graders (Probabilistic)

  • Known as "LLM-as-a-Judge"
  • Use strong models like Gemini 3 Pro
  • Measure nuance, reasoning, helpfulness, safety
  • Advantages: Evaluate complex, open-ended tasks
  • Limitations: Slower, expensive, requires careful prompt engineering

Vertex AI Evaluation Metrics

The Vertex AI Gen AI Evaluation Service provides managed metrics including:

Rubric-based metrics (incorporate LLMs):

  • Adaptive rubrics: Dynamically generated per prompt with granular, explainable feedback
  • Static rubrics: Explicitly defined, applied uniformly with numerical scores

Computation-based metrics: Deterministic algorithms using ground truth

Custom function metrics: User-defined Python functions

Specific Metrics for This Lab

Understanding Safe Deployment

Revision Tags and Shadow Deployment

Google Cloud Run creates immutable revisions with every deployment. Tags enable access to specific revisions via special URLs while receiving zero public traffic — essential for safe testing.

Why Not Local Testing?

While ADK supports local evaluation, deploying to hidden revisions provides critical advantages:

  • Environment Parity: Testing in actual cloud runtime ensures compatibility (network, CPU, memory, secrets)
  • Multi-Agent Interaction: Distributed systems communicate over HTTP; shadow deployment tests real network latency and authentication
  • Secrets & Permissions: Verifies service account permissions function correctly for Vertex AI, Firestore, etc.
📌
This represents Proactive Evaluation (pre-deployment checks). Post-deployment, Reactive Monitoring (observability) catches production issues.

CI/CD Workflow: Deploy, Assess, Promote

The robust continuous deployment pipeline follows:

  1. Commit: Code change pushed to repository
  2. Deploy (Hidden): New revision tagged with commit hash (e.g., c-abc1234), receiving 0% traffic
  3. Evaluate: Evaluation script targets specific revision URL (e.g., https://c-abc1234---researcher-xyz.run.app)
  4. Promote: After evaluation passes, traffic migrates to new revision
  5. Rollback: Failed revisions never reach users; bad versions simply remain undeployed

Analyzing evaluate.sh

The script automates the process:

Bash
export COMMIT_SHORT_HASH=$(git rev-parse --short HEAD)
export COMMIT_REVISION_TAG="c-${COMMIT_SHORT_HASH}"

source ./deploy.sh --revision-tag $COMMIT_REVISION_TAG --no-redeploy
uv run -m evaluator.evaluate_agent

The deploy.sh handles revision deployment with --no-traffic and --tag options. Existing running services remain unaffected. The new hidden revision receives no traffic unless explicitly called using its revision tag URL.

Implement the Evaluation Script

Define the Metrics

For the Researcher Agent with golden answers (ground truth), measuring Capability Evaluation:

  • Final Response Match: Reference-based. Does answer match expected answer semantically and factually?
  • Tool Use Quality: Adaptive rubrics. Did agent select appropriate tools with correct parameters in proper sequence?
  • Tool Use Trajectory: Custom metrics measuring tool usage precision and recall against expected sequences

Custom Tool Use Trajectory Metrics

Custom metrics use Python functions in shared/evaluation/tool_metrics.py. The Vertex AI Evaluation Service executes these in a sandbox environment.

The get_custom_function_metric helper converts Python functions to custom code evaluation metrics by:

  1. Extracting module source code
  2. Creating an evaluate function wrapping the original function
  3. Returning an EvaluationRunMetric object with CustomCodeExecutionSpec
Python
import inspect
module_source = inspect.getsource(
    inspect.getmodule(metrics_function)
)
module_source += (
    "\n\ndef evaluate(instance: dict) -> float:\n"
    f"    return {metrics_function.__name__}(instance)\n"
)

Add Metrics and Evaluation Code

Add to evaluator/evaluate_agent.py after if __name__ == "__main__":

Python
eval_data_researcher = os.path.dirname(__file__) + "/eval_data_researcher.json"
metrics=[
    # Compares the agent's output against a "Golden Answer"
    types.RubricMetric.FINAL_RESPONSE_MATCH,
    # Did the agent use the tools effectively?
    types.RubricMetric.TOOL_USE_QUALITY,
    # Custom metrics for tools trajectory analysis
    get_custom_function_metric("trajectory_precision", trajectory_precision_func),
    get_custom_function_metric("trajectory_recall", trajectory_recall_func)
]

print("🧪 Running Researcher Evaluation...")
eval_results = asyncio.run(
    evaluate_agent(
        agent_api_server=RESEARCHER_URL,
        agent_name="agent",
        evaluation_data_file=eval_data_researcher,
        evaluation_storage_uri=f"gs://{GOOGLE_CLOUD_PROJECT}-agents/evaluation",
        metrics=metrics,
        project_id=GOOGLE_CLOUD_PROJECT,
        location=GOOGLE_CLOUD_REGION
    )
)
print(f"\n🧪 Researcher Evaluation results:\n{eval_results}")
print(f"Evaluation Run ID: {eval_results.run_id}")

Add Evaluation Success Criteria and Gating

In production, implement Evaluation Success Criteria and Gating Steps to prevent bad revisions from receiving traffic. Append gating logic:

Python
METRIC_THRESHOLD = 0.75
researcher_eval_failed = False
if eval_results.state != types.EvaluationRunState.SUCCEEDED:
    print(f"🛑 Researcher Evaluation failed with state {eval_results.state}.")
    researcher_eval_failed = True
else:
    for metric_name, metric_values in eval_results.metrics.items():
        if metric_values["mean"] < METRIC_THRESHOLD:
            print(f"🛑 Researcher Evaluation failed with metric `{metric_name}` below {METRIC_THRESHOLD} threshold.")
            researcher_eval_failed = True
if researcher_eval_failed:
    exit(1)
⚠️
When any evaluation metric mean falls below the 0.75 threshold, deployment fails.

Inspect Evaluation Data

Two evaluation data files exist in evaluator/:

Understand the Evaluation Code

Opening shared/evaluation/evaluate.py reveals the core logic with function evaluate_agent:

Process Steps

  1. Data Loading: Reads evaluation dataset from file
  2. Parallel Inference: Runs agent against dataset in parallel, capturing final response and intermediate tool execution trace
  3. Vertex AI Evaluation: Merges original evaluation data with responses and trace, submitting to Vertex AI Evaluation Service with GenAI Client in Vertex AI SDK

The critical moment calls create_evaluation_run:

Python
evaluation_run = client.evals.create_evaluation_run(
    dataset=agent_dataset_with_inference,
    agent_info=agent_info,
    metrics=metrics,
    dest=evaluation_storage_uri
)

The Agent Info API

For accurate evaluation, the Evaluation Service requires agent configuration including system instructions, description, and available tools, passed as the agent_info parameter.

Opening shared/adk_app.py and searching for def agent_info reveals the ADK application exposes a helper endpoint:

Python
@app.get("/apps/{agent_name}/agent-info")
async def agent_info(agent_name: str) -> typing.Dict[str, typing.Any]:
    return {
        "name": agent.name,
        "instruction": str(getattr(agent, "instruction", None)),
        "tool_declarations": tools_dict_list
    }
💡
This endpoint (enabled via --publish_agent_info flag) allows evaluation scripts to dynamically fetch runtime agent configuration — crucial for metrics assessing tool usage since the judge model evaluates more effectively when knowing specifically available tools.

Retrieve and Display Results

Find the Retrieve Evaluation Run and Display Results cell and add:

from vertexai import Client

client = Client(
    project=GOOGLE_CLOUD_PROJECT,
    location=GOOGLE_CLOUD_REGION,
    http_options=genai_types.HttpOptions(api_version="v1beta1"),

    name=EVAL_RUN_ID,
    include_evaluation_items=True

Interpreting Results

  • Regression vs Capability: Did score drop on old tests (regression — investigate) or improve on new tests (capability — progress)?
  • Failure Analysis: Examine the trace, not just score. Did it call wrong tool? Failed parsing? Review judge LLM explanations and verdicts.
  • Pass@1 vs Pass@k: Single test runs give Pass@1 score. Non-determinism may cause failures. Sophisticated setups run each test k times. Many metrics do this internally; for example, "Final Response Match" makes 5 judge LLM calls.

Continuous Integration and Deployment (CI/CD)

Production systems should run agent evaluation in CI/CD pipelines. Google Cloud Build is suitable.

For every repository commit, evaluation runs with other tests. Passing evaluation enables deployment promotion to serve users. Failures maintain status quo while developers investigate.

Cloud Build Configuration

Create cloudbuild.yaml:

YAML
steps:
    - name: gcr.io/google.com/cloudsdktool/google-cloud-cli:latest
      entrypoint: /bin/bash
      env:
          - 'BUILD_ID=$BUILD_ID'
      args:
          - "-c"
          - |
            if [[ "$_COMMIT_SHORT_HASH" != "" ]]; then
              export COMMIT_SHORT_HASH=$_COMMIT_SHORT_HASH
            else
              export COMMIT_SHORT_HASH=$SHORT_SHA
            fi
            export COMMIT_REVISION_TAG="c-$${COMMIT_SHORT_HASH}"
            echo "Deploying with revision tag: $$COMMIT_REVISION_TAG"
            set -e
            curl -LsSf https://astral.sh/uv/install.sh | sh
            source $$HOME/.local/bin/env
            uv sync

            source ./deploy.sh --revision-tag $$COMMIT_REVISION_TAG --no-redeploy
            uv run -m evaluator.evaluate_agent

            echo "Promoting revisions $$COMMIT_REVISION_TAG to serve 100% of traffic."
            gcloud run services update-traffic researcher --to-tags $$COMMIT_REVISION_TAG=100 --region $$GOOGLE_CLOUD_REGION --project $$GOOGLE_CLOUD_PROJECT
            gcloud run services update-traffic judge --to-tags $$COMMIT_REVISION_TAG=100 --region $$GOOGLE_CLOUD_REGION --project $$GOOGLE_CLOUD_PROJECT
            gcloud run services update-traffic content-builder --to-tags $$COMMIT_REVISION_TAG=100 --region $$GOOGLE_CLOUD_REGION --project $$GOOGLE_CLOUD_PROJECT
            gcloud run services update-traffic orchestrator --to-tags $$COMMIT_REVISION_TAG=100 --region $$GOOGLE_CLOUD_REGION --project $$GOOGLE_CLOUD_PROJECT
            gcloud run services update-traffic course-creator --to-tags $$COMMIT_REVISION_TAG=100 --region $$GOOGLE_CLOUD_REGION --project $$GOOGLE_CLOUD_PROJECT

options:
  substitutionOption: 'ALLOW_LOOSE'
  defaultLogsBucketBehavior: REGIONAL_USER_OWNED_BUCKET

Running the Pipeline

Create a Cloud Build service account. Create run_cloud_build.sh:

Bash
#!/bin/bash

set -e
source .env

BUILD_SA_NAME="agent-eval-build-sa"
BUILD_SA_EMAIL="${BUILD_SA_NAME}@${GOOGLE_CLOUD_PROJECT}.iam.gserviceaccount.com"
COMMIT_SHORT_HASH=$(git rev-parse --short HEAD)

if ! gcloud iam service-accounts describe "${BUILD_SA_EMAIL}" --project "${GOOGLE_CLOUD_PROJECT}" &> /dev/null; then
    echo "Creating service account ${BUILD_SA_NAME} for Cloud Build."
    gcloud iam service-accounts create ${BUILD_SA_NAME} --project "${GOOGLE_CLOUD_PROJECT}" --display-name "Agent Build Service Account"

    echo "Granting roles to service account ${BUILD_SA_NAME}..."
    sleep 10
    ROLES=(
        "roles/cloudbuild.builds.builder"
        "roles/run.admin"
        "roles/run.invoker"
        "roles/iam.serviceAccountOpenIdTokenCreator"
        "roles/iam.serviceAccountUser"
        "roles/serviceusage.serviceUsageAdmin"
        "roles/serviceusage.serviceUsageConsumer"
        "roles/aiplatform.user"
        "roles/logging.logWriter"
        "roles/storage.admin"
        "roles/artifactregistry.writer"
    )

    for ROLE in "${ROLES[@]}"; do
        gcloud projects add-iam-policy-binding "$GOOGLE_CLOUD_PROJECT" \
            --member="serviceAccount:$BUILD_SA_EMAIL" \
            --role="$ROLE" \
            --condition=None \
            --quiet
    done

    echo "Waiting for 60 seconds for the permission changes to propagate..."
    sleep 60
fi

gcloud builds submit --config cloudbuild.yaml \
    --service-account="projects/${GOOGLE_CLOUD_PROJECT}/serviceAccounts/${BUILD_SA_EMAIL}" \
    --machine-type=e2-highcpu-32 \
    --timeout=120m \
    --substitutions _COMMIT_SHORT_HASH=$COMMIT_SHORT_HASH,_GOOGLE_CLOUD_PROJECT=$GOOGLE_CLOUD_PROJECT,_GOOGLE_CLOUD_LOCATION=$GOOGLE_CLOUD_LOCATION,_GOOGLE_CLOUD_REGION=$GOOGLE_CLOUD_REGION

Run the pipeline:

Shell
chmod +x run_cloud_build.sh
./run_cloud_build.sh
💡
In production, set up a Cloud Build Trigger to execute automatically on every git push, ensuring every commit goes through this evaluation workflow.
